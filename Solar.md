# 真实比例太阳系模拟网站需求文档

## 1. 项目概述

### 1.1 项目名称

暂定名称：**True Scale Solar System / 真实比例太阳系**

### 1.2 项目目标

开发一个基于 Web 的交互式三维太阳系模拟网站，按照真实天文数据展示：

- 太阳与八大行星的相对尺寸；
- 各天体之间的相对距离；
- 行星真实的公转周期；
- 天体真实的自转周期及自转方向；
- 行星轨道形状、倾角和运行方向；
- 指定时间下各天体的大致位置。

天体表面不需要写实材质，只需使用颜色不同、尺寸符合比例的球体表达。

网站的主要价值是让用户直观理解：

1. 太阳系内部尺度有多大；
2. 行星与太阳相比有多小；
3. 行星之间有多空旷；
4. 不同行星自转与公转速度差异有多大；
5. 真实比例与传统太阳系示意图之间的巨大差别。

## 2. 核心设计原则

### 2.1 数据真实，展示可切换

系统内部必须始终保存真实数据，展示层允许用户切换不同模式：

- 严格真实比例；
- 增强可见比例；
- 教学演示比例。

任何非真实缩放必须明确显示提示，不允许让用户误以为当前画面仍是完全真实比例。

### 2.2 尺寸比例与距离比例相互独立

真实太阳系中，行星尺寸远小于行星间距。在完全真实比例下，从太阳系整体视角几乎无法看到行星。

因此必须提供两个独立控制项：

- `天体尺寸倍率`
- `轨道距离倍率`

默认真实模式下均为 `1×`。

用户放大行星尺寸时，只改变球体显示尺寸，不改变轨道距离和运动计算。

### 2.3 时间必须连续

模拟时间不能基于累计帧数计算，而应基于统一的模拟时间戳计算。

无论用户：

- 暂停；
- 调整速度；
- 切换标签页；
- 浏览器掉帧；
- 改变摄像机位置；

天体位置都应由当前模拟时间直接计算，避免逐帧累加产生误差。

### 2.4 明确区分“示意模拟”与“高精度星历”

首个版本可以采用开普勒轨道模型，根据轨道根数计算天体位置。

后续高精度模式可接入 JPL Horizons 星历数据。JPL 明确指出，平均轨道参数适合描述轨道形态，但高精度位置计算应使用 Horizons 星历服务。([ssd.jpl.nasa.gov](https://ssd.jpl.nasa.gov/orbits.html?utm_source=chatgpt.com))

## 3. 用户与使用场景

### 3.1 目标用户

- 对天文学感兴趣的普通用户；
- 中小学科学教育用户；
- 科普内容创作者；
- 希望理解太阳系真实尺度的用户；
- 希望观察行星运动规律的开发者或研究爱好者。

### 3.2 典型场景

#### 场景一：观察真实尺度

用户从太阳附近开始，保持尺寸和距离均为真实比例，通过高速移动前往地球、木星和海王星，感受太阳系的巨大空间。

#### 场景二：比较天体大小

用户进入“尺寸比较”视图，将太阳和八大行星排列在同一画面中，对比直径，但不表现真实间距。

#### 场景三：观察公转速度

用户放大模拟时间，观察水星快速绕太阳运行、海王星缓慢移动。

#### 场景四：观察自转

用户聚焦某颗行星，提高时间倍率，观察其自转速度、自转轴方向以及顺行或逆行特征。

#### 场景五：查看指定日期

用户输入日期和时间，系统将太阳系定位到对应模拟时刻。

## 5. 天文数据模型

### 5.1 每个天体的数据结构

建议定义统一的 `CelestialBody` 数据模型：

```ts
interface CelestialBody {
  id: string;
  name: string;
  localizedName: string;
  type: "star" | "planet" | "dwarf_planet" | "moon";

  radiusKm: number;
  massKg?: number;

  rotationPeriodHours: number;
  axialTiltDeg: number;
  rotationDirection: "prograde" | "retrograde";

  orbit?: {
    parentId: string;

    semiMajorAxisKm: number;
    eccentricity: number;
    inclinationDeg: number;

    longitudeOfAscendingNodeDeg: number;
    argumentOfPeriapsisDeg: number;
    meanAnomalyAtEpochDeg: number;

    orbitalPeriodDays: number;
    epoch: string;
  };

  display: {
    color: string;
    emissive?: boolean;
    hasRing?: boolean;
  };

  dataSource: string;
}
```

### 5.2 数据来源

物理参数优先采用：

1. NASA/JPL Solar System Dynamics；
2. NASA Planetary Fact Sheets；
3. NASA Solar System Exploration；
4. IAU 标准或报告。

JPL 的行星物理参数表包含行星及矮行星的选定物理特征；NASA Planetary Fact Sheets 提供体积、轨道和其他基础参数。([ssd.jpl.nasa.gov](https://ssd.jpl.nasa.gov/planets/phys_par.html?utm_source=chatgpt.com))

NASA 的行星比较数据包含行星自转周期和公转周期，可用于交叉校验基础配置。([NASA Science](https://solarsystem.nasa.gov/planet-compare/?utm_source=chatgpt.com))

### 5.3 数据单位

内部统一使用：

- 距离：千米 `km`
- 半径：千米 `km`
- 时间：秒 `s`
- 角度：弧度 `rad`
- 模拟时刻：UTC
- 质量：千克 `kg`

渲染前再转换为引擎世界坐标。

禁止在核心计算中混用：

- 天文单位；
- 千米；
- 米；
- 地球日；
- 儒略日；
- 浏览器毫秒时间戳。

### 5.4 自转周期符号

推荐将自转周期存为绝对值，同时用 `rotationDirection` 表示方向。

不要仅依赖负数表示逆行，否则容易在 UI、计算和数据导入过程中产生歧义。

需要正确表达：

- 金星逆向自转；
- 天王星的特殊自转方向和极大轴倾角；
- 其他行星通常表现为顺行自转。

NASA 对天王星的说明指出，其自转轴相对于轨道平面接近侧躺状态。([NASA Science](https://science.nasa.gov/uranus/facts/?utm_source=chatgpt.com))

## 6. 比例系统

### 6.1 世界坐标比例

建议定义：

```ts
worldUnits = physicalKilometers / distanceScaleKmPerUnit;
```

例如：

```ts
1 world unit = 100,000 km
```

该值仅用于避免 WebGL 坐标过大，不代表用户看到的比例被改变。

### 6.2 尺寸计算

```ts
renderRadius =
  physicalRadiusKm /
  distanceScaleKmPerUnit *
  bodySizeMultiplier;
```

其中：

- 真实模式：`bodySizeMultiplier = 1`
- 增强模式：用户可设为 `10×、100×、1000×`
- 自适应模式：根据摄像机距离提供最低屏幕像素尺寸

### 6.3 距离计算

```ts
renderPosition =
  physicalPositionKm /
  distanceScaleKmPerUnit *
  orbitDistanceMultiplier;
```

真实模式下：

```ts
orbitDistanceMultiplier = 1;
```

### 6.4 三种比例模式

#### A. 严格真实模式

- 天体尺寸倍率：`1×`
- 轨道距离倍率：`1×`
- 不设置最低可见尺寸
- 用户可能只能看到像素级行星
- 提供方向标记和定位功能，但不改变球体尺寸

#### B. 增强可见模式

- 轨道距离保持真实比例
- 行星球体进行统一倍率放大
- 默认可使用 `100×` 或动态倍率
- 页面顶部持续显示：

> 当前天体尺寸已放大 100 倍，轨道距离保持真实比例。

#### C. 教学演示模式

- 天体尺寸和轨道距离分别压缩
- 目标是让大部分行星同时出现在画面中
- 必须明确标记为非真实比例

## 7. 时间模拟系统

### 7.1 模拟时间

核心变量：

```ts
simulationTimeUtc: number;
timeScale: number;
paused: boolean;
```

时间更新：

```ts
simulationTimeUtc +=
  realDeltaSeconds * timeScale;
```

但天体位置应从 `simulationTimeUtc` 直接求值，而不是在上一帧位置基础上累加。

### 7.2 时间倍率

建议提供：

- 实时 `1×`
- `1 分钟/秒`
- `1 小时/秒`
- `1 天/秒`
- `10 天/秒`
- `30 天/秒`
- `1 年/秒`
- 自定义倍率
- 反向时间

内部可将倍率统一表示为：

```ts
模拟秒 / 现实秒
```

例如：

```text
1×        = 1
1天/秒    = 86,400
1年/秒    = 31,557,600
```

### 7.3 时间控制组件

必须包含：

- 播放；
- 暂停；
- 回到当前时间；
- 时间倍率选择；
- 正向或反向；
- 日期时间输入；
- 单步推进；
- 当前模拟时间显示；
- UTC与用户本地时间切换。

### 7.4 自转计算

天体自转角：

```ts
rotationAngle =
  initialRotationAngle +
  directionSign *
  2 * Math.PI *
  elapsedSeconds /
  rotationPeriodSeconds;
```

其中：

```ts
directionSign = prograde ? 1 : -1;
```

为了观察自转，球体即使没有真实纹理，也需要添加一种能体现方向的视觉特征，例如：

- 经线；
- 本初子午线；
- 赤道线；
- 半球明暗差异；
- 球面箭头；
- 表面固定标记点。

否则纯色球体旋转时用户无法察觉。

### 7.5 公转计算

MVP 不应简单使用匀速圆周运动。

至少需要按照开普勒椭圆轨道计算：

1. 根据时间计算平近点角；
2. 求解开普勒方程；
3. 得到偏近点角；
4. 计算轨道平面内坐标；
5. 根据升交点经度、轨道倾角和近心点幅角旋转至三维空间。

基本过程：

```text
M = M₀ + n × Δt
M = E - e × sin(E)

x = a × (cos(E) - e)
y = a × sqrt(1 - e²) × sin(E)
```

其中：

- `a`：半长轴；
- `e`：轨道离心率；
- `M`：平近点角；
- `E`：偏近点角；
- `n`：平均角速度。

开普勒方程可采用 Newton-Raphson 迭代求解。

### 7.6 精度等级

#### MVP：教育级近似

- 固定轨道根数；
- 开普勒椭圆轨道；
- 不计算行星摄动；
- 不计算质心运动；
- 太阳固定在场景原点；
- 适合科普和尺度展示。

#### 高级：星历级位置

- 使用 JPL Horizons 返回指定时刻的位置；
- 使用日心或太阳系质心坐标；
- 支持历史与未来时间；
- 对星历数据进行服务端缓存。

JPL Horizons 提供太阳系天体随时间变化的位置和运动数据，是高精度模式的合适数据源。([ssd.jpl.nasa.gov](https://ssd.jpl.nasa.gov/horizons/manual.html?utm_source=chatgpt.com))

## 8. 三维场景

### 8.1 推荐技术

建议使用：

- TypeScript
- React 或 Next.js
- Three.js
- React Three Fiber
- Zustand
- Web Worker
- WebGL 2

不建议在首版直接使用完整物理引擎。行星轨道应由确定性数学公式驱动，而不是刚体或引力物理模拟。

### 8.2 天体渲染

所有天体使用基础球体：

- 不使用真实照片纹理；
- 每颗天体使用不同纯色；
- 太阳使用自发光材质；
- 行星使用基础漫反射或简化受光材质；
- 球体网格精度根据屏幕尺寸动态选择；
- 很远或很小的天体降低分段数。

### 8.3 光照

建议：

- 太阳作为点光源；
- 使用低强度环境光，避免行星背面完全不可见；
- 提供“科学光照”和“辅助光照”开关。

科学光照：

- 主要由太阳照明；
- 保留明显昼夜面。

辅助光照：

- 提高暗面可见性；
- 明确标记为视觉辅助。

### 8.4 背景

默认使用纯黑或极深色背景。

可选：

- 稀疏星点；
- 无背景星空；
- 网格；
- 黄道面；
- 坐标轴。

星点仅作为视觉背景，不得让用户误以为星点与太阳系处于相同比例距离。

### 8.5 轨道线

轨道线必须：

- 根据椭圆轨道生成；
- 体现离心率；
- 体现轨道倾角；
- 可单独开关；
- 聚焦天体时高亮对应轨道；
- 支持调整透明度。

禁止默认将所有轨道画成同心圆。

## 9. 摄像机与导航

### 9.1 摄像机模式

#### 自由浏览

- 鼠标拖动旋转；
- 滚轮缩放；
- 右键或组合键平移；
- 键盘进行空间移动。

#### 天体跟随

摄像机锁定某个天体：

- 跟随天体公转；
- 用户仍可绕天体旋转；
- 支持固定观察距离；
- 可选择是否随天体自转。

#### 俯视模式

- 从黄道面法线方向观察；
- 适合查看整体轨道。

#### 表面附近模式

- 摄像机移动到天体附近；
- 展示其相对大小和自转；
- 不要求实现真实地表降落。

### 9.2 快速定位

提供天体列表：

- 太阳；
- 八大行星。

点击后：

1. 平滑移动摄像机；
2. 聚焦目标；
3. 显示天体信息；
4. 高亮轨道；
5. 自动调整近裁剪面和远裁剪面。

### 9.3 距离移动辅助

真实太阳系跨度很大，普通固定速度移动会导致：

- 靠近行星时太快；
- 星际空间中太慢。

因此移动速度应支持：

- 自动指数级调整；
- 手动挡位；
- 根据摄像机与目标距离改变；
- 长距离“跃迁至目标”。

建议挡位：

```text
1 km/s
100 km/s
10,000 km/s
0.01 AU/s
0.1 AU/s
1 AU/s
```

## 10. 数值精度与大尺度渲染

### 10.1 浮点精度问题

太阳系尺度远大于行星半径，直接将所有坐标作为普通单精度 GPU 坐标传入会造成：

- 靠近远距离行星时球体抖动；
- 顶点精度下降；
- 轨道线断裂；
- 摄像机移动不稳定。

### 10.2 推荐解决方案

采用“浮动原点”：

- 天文位置以 JavaScript `number` 或更高精度形式保存；
- 每帧将当前摄像机或目标天体设为局部原点；
- 传入 GPU 的位置为：

```ts
localPosition = bodyAbsolutePosition - cameraOrigin;
```

不要通过不断修改全局天体位置来模拟移动。

### 10.3 裁剪面

近裁剪面和远裁剪面需要动态调整。

同时考虑：

- 对数深度缓冲；
- 分层场景；
- 局部摄像机；
- 轨道线独立渲染；
- 远景标记使用屏幕空间 UI。

## 12. 比例体验功能

### 12.1 尺寸比较视图

此视图独立于轨道场景：

- 将太阳和八颗行星横向排列；
- 所有球体使用相同尺寸比例；
- 忽略真实间距；
- 提供太阳是否显示开关；
- 支持按直径或体积排序。

必须显示：

> 当前视图仅比较天体尺寸，不代表天体之间的实际距离。

### 12.2 距离漫游视图

用户可从太阳向外移动：

- 太阳；
- 水星轨道；
- 金星轨道；
- 地球轨道；

## 17. 错误处理

需要处理：

- 星历接口不可用；
- 数据加载失败；
- WebGL 不可用；
- GPU 性能过低；
- 日期超出星历支持范围；
- URL 参数非法；
- 用户输入极端时间倍率；
- 天体尺寸倍率过大导致遮挡；
- 摄像机进入球体内部。

错误处理原则：

- 高精度星历不可用时，自动回退到开普勒近似；
- 回退后明确显示当前模型；
- 不因为外部接口不可用而导致网站完全无法运行；
- MVP 基础数据应随前端静态资源打包。

## 20. 验收标准

### 20.1 比例验收

- 当尺寸倍率和距离倍率均为 `1×` 时，所有球体半径和轨道半长轴使用同一个世界坐标换算比例；
- 不允许为某一颗行星单独设置视觉放大；
- 增强模式必须显示倍率；
- 尺寸对比模式必须标记“不代表真实距离”。

### 20.2 运动验收

- 模拟运行一个地球年后，地球应大致完成一周公转；
- 同一时间内，各行星完成的公转比例应符合其公转周期；
- 自转角速度应与自转周期成反比；
- 金星和天王星的旋转方向需要正确；
- 暂停后所有公转和自转停止；
- 恢复后从同一模拟时间继续；
- 修改帧率不能改变模拟结果。

### 20.3 轨道验收

- 轨道不是统一同心圆；
- 行星位置位于自己的轨道曲线上；
- 轨道倾角正确应用；
- 近日点附近速度高于远日点附近；
- 指定同一日期时，刷新页面后位置保持一致。

### 20.4 精度验收

MVP 应与选定基准数据在合理误差范围内一致。

建议规定：

- 半径误差：小于 `0.01%`
- 公转周期误差：小于 `0.01%`
- 自转周期误差：小于 `0.01%`
- 半长轴误差：小于 `0.01%`
- 同一固定时刻重复计算位置结果完全一致
- 长时间运行不依赖帧数累积

### 20.5 交互验收

- 用户可以在三次操作以内聚焦任意行星；
- 用户不会因为进入真实比例模式而完全失去方向；
- 用户可以一键回到太阳系整体视图；
- 用户可以一键回到当前时间；
- 摄像机不会轻易穿入天体内部；
- 移动端可以完成旋转、缩放、选择和时间控制。

## 23. 给开发 Agent 的关键限制

1. 不要用匀速同心圆代替行星轨道。
2. 不要让公转速度和浏览器帧率绑定。
3. 不要把尺寸倍率同时应用到轨道距离。
4. 不要在真实比例模式中偷偷设置行星最小尺寸。
5. 不要只旋转纯色球体而不给用户观察自转的视觉标记。
6. 不要依赖完整物理引擎计算公转。
7. 不要把所有天体直接放在超大绝对 GPU 坐标中。
8. 不要默认宣称开普勒近似是某一时刻的精确星历。
9. 不要在外部星历接口失败时让整个应用不可用。
10. 不要将土星环、背景星空或轨道线误算入真实天体尺寸比例。
11. 所有倍率、单位和模型状态必须对用户透明。
12. 所有核心天文数据必须记录来源和版本。
